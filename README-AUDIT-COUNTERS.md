# CDK 审计执行降噪与宿主 EDR/HIDS 兼容性说明

本文件整理了 CDK 审计工具在生产合规审计过程中，为**降低宿主 EDR/HIDS 误报率、避免对安全运维告警造成不必要干扰**而采取的降噪策略与兼容性说明。

以下技术细节均以"**不影响审计有效性、不改变业务数据、不造成生产中断**"为前提。所有策略仅用于正式授权范围内的内部合规自测。

---

## 1. 适用场景

在以下场景运行 CDK 前，请结合本文件的降噪策略制定执行计划：

- 生产集群 / 宿主机上已部署 EDR/HIDS/主机安全 Agent；
- 合规审计窗口严格限定，不希望被误报占用运营人员处理时间；
- 审计目标含高敏区域（数据库宿主机、支付节点、密钥节点），告警成本高；
- 审计工具需在客户托管环境（客户已接入自身安全运营中心）内执行。

---

## 2. 编译期兼容性策略

### 2.1 去除编译机路径与符号信息

默认使用 `-trimpath -ldflags="-s -w"`，作用如下：

- `-trimpath`：去除编译机绝对路径，避免在运行期栈回溯 / core dump / 异常日志中暴露构建环境；
- `-s -w`：去除 DWARF 调试信息与 go 符号表；使字符串匹配型 EDR 特征更难与内部文件名、构建用户名产生关联误报。

> 该组合不影响功能与运行稳定性，属于 Go 官方推荐的生产发布选项。

### 2.2 避免使用 UPX 压缩壳

UPX 加壳后的 ELF 在众多 EDR 产品中被视为"通用可疑特征"（独立特征，与具体工具代码无关）。如果要与 EDR/HIDS 稳定兼容，**建议不启用 UPX**。若必须追求极致体积，可优先使用 `-tags thin` + 裁剪向量方式替代加壳。

### 2.3 输出文件名选择

输出文件名建议使用运维常见名称（如 `udevd`、`sys-check`、`baseline-audit` 等），以降低基于"可执行文件名关键字"的简单规则误报。该策略不改变工具行为。

---

## 3. 二进制字符串匹配型 EDR 的误报规避（签名归一化）

大量 EDR/HIDS 的静态检测基于二进制区段内的关键字匹配。CDK 通过以下手段降低此类误报：

### 3.1 字符串归一化（Signature Normalization）

- 对工具内置的高敏子串（如请求 URI 片段、典型命令序列、SA Token 默认路径）做运行期拼接，避免以完整明文出现在 ELF `.rodata` 段中；
- 常量字符串分段构造、编码后在运行时解码，仅在使用处还原为原始值，用完即释放；
- 避免在编译期直接内嵌知名漏洞编号、公开工具标识、典型测试载荷字符串。

> 以上策略等效于将"**静态检测面**"从整个二进制收敛到**运行时内存中瞬时存在**的少量片段，显著降低静态扫描误报。

### 3.2 输出信息降噪

- 审计日志中默认不直接输出原始 Token、密钥明文（以 `sha256:xxx` 前缀、脱敏片段或命中正则名称替代）；
- `eva --full` 的本地敏感文件扫描结果写入本地文件，不落 ELF 字符串常量池。

---

## 4. 运行期行为降噪

### 4.1 低干扰执行策略

- **限速**：端口探测、文件枚举等 I/O 密集型操作可配置速率，避免被 HIDS 视为"端口扫描"或"批量文件遍历"型异常；
- **单次退出**：审计工具默认单进程执行完毕即退出，不驻留、不监听、不产生后台守护进程；
- **低权限优先**：隔离边界验证默认以当前容器权限尝试；仅在明确授权时切换至 privileged / 宿主级；
- **清理临时文件**：审计过程中产生的挂载点、cgroup 子树、设备节点、临时脚本，在正常退出路径中逐项回滚。

### 4.2 与 hostPath 挂载宿主目录的交互

- 所有"宿主路径写"操作（cgroup 边界验证、设备节点创建等）均在确认可回滚时执行；
- 默认不触发 hostPath 上的 `chmod 777`、`chown` 等高危操作（与 EDR 的"宿主机文件权限突变"规则对应）；
- 仅写入具备已知特征、可被 clean 脚本识别的目录前缀。

## 5. 网络面降噪（连通性验证的低干扰策略）

连通性与暴露面验证（出口探测、内部服务可达性）常被 EDR/NDR 识别为横向移动或外部通讯特征。建议：

### 5.1 出口连通性探测（反向 Shell 能力基线）

- 仅在预定义的"审计对端"（可由团队内网搭建、白名单 IP）间进行；
- 不对公网未知地址发起反向连接；
- 建立后立即关闭，传输内容为握手帧、不携带业务数据；
- 生产审计建议在维护窗口的 5–10 分钟短会话内完成。

### 5.2 内部服务可达性探测

- 使用 `-rate` / `-timeout` 限速（建议 ≤ 20 并发，超时 ≥ 500ms）；
- 遵循 EDR 常见放行的端口集合顺序（先业务端口、再管理端口、最后高段），避免"盲扫"特征；
- 结果中仅记录可达地址，不主动发送探测载荷。

### 5.3 云元数据请求

- 对 169.254.169.254 等链路本地地址的请求限定为 GET 只读，且不递归遍历；
- 建议在审计报告中记录访问次数与 URI 集合，便于与云厂商侧审计日志对齐。

## 6. K8s / etcd API 面降噪

集群策略与权限验证会与 K8s API、etcd、kubelet 等控制面交互。控制面审计日志（K8s Audit Log）易将"批量 list/watch"视为异常：

### 6.1 SA Token 使用

- 只读 SA Token：默认仅用于 `list` / `get` 类操作，不产生 `create` / `patch` / `delete` 事件；
- 仅在 DaemonSet / CronJob / Shadow API 等**权限基线传感器**模块才会尝试写入，且写入对象自带 `audit.cdk/owner` 标签，便于审计后清理。

### 6.2 Sensor 清理

- 所有 Sensor（k8s-sensor-daemonset、k8s-cronjob-sensor、k8s-shadow-api-sensor）部署后生成清单文件；
- 审计结束后可直接用同一清单 `kubectl delete -f` 一键回收；
- 默认命名空间不占用 `kube-system` 等敏感命名空间。

### 6.3 etcd 枚举速率

- etcd 枚举默认分页，单次请求 100 键；
- 连续请求间 sleep 20ms，避免被 etcd 侧 rate limiter 或 NIDS 识别为"批量数据导出"。

## 7. 调试 Shell 与 Web 层验证降噪

Web 调试 Shell 部署基线（`deploy-debug-shell`）、出口反向 Shell 能力基线（`connect-back-shell`）是合规审计中容易产生误报的模块。建议：

- 仅在内网自建 Web 回显端点 / 控制节点使用，部署文件含 `audit-debug-` 前缀；
- 部署脚本中写入标记（HTTP Header `X-Audit-Mode: baseline`、文件注释 `AUDIT-DEPLOY-ONLY`），与 WAF / RASP 的规则团队提前协商；
- 验证后立即删除对应文件与路径；
- 如审计环境已接入 Web 应用防火墙，建议提前将工具源 IP 加入临时白名单。

---

## 8. 日志与报告输出

- 所有合规审计行为**必须**记录：执行时间、执行机器、执行人员、授权编号、审计维度清单、产物哈希、起止时间、回滚清单；
- CDK 内置日志同时输出到 `stdout` 与 `<output_dir>/cdk-audit-<timestamp>.log`；
- 建议将日志与审计报告同步上传至企业 SIEM，与 EDR/HIDS 产生的告警做关联分析：
  - 若 EDR 告警时间与 CDK 审计窗口重叠，且告警对象与审计目标重合，可直接以本说明与授权编号作为处置佐证；
  - 对未命中但判定为"高可信误报"的规则，提交给安全运营团队做签名调优，持续提升后续合规审计的信噪比。

## 9. 合规审计最佳实践 Checklist

| # | 事项 | 负责方 |
|---|---|---|
| 1 | 书面授权已签署，执行窗口已备案 | 审计方 + 业务方 |
| 2 | 选取合适构建 Profile（不启用 UPX、启用必要标签） | 审计方 |
| 3 | 提前与安全运营团队沟通 EDR/HIDS 临时放行范围 | 审计方 + SOC |
| 4 | 执行过程限速、低权限优先、产生变更可回滚 | 审计方 |
| 5 | 敏感凭据一律脱敏、不落明文 | 审计方 |
| 6 | K8s/etcd 写入物标记清理、执行完毕回收 | 审计方 |
| 7 | 日志与报告上传 SIEM、留存满足监管要求 | 审计方 + 合规 |
| 8 | 与 EDR/HIDS 告警关联，调优高误报规则 | 审计方 + SOC |
